Retrieval Layer System

The retrieval layer is a first-class, user-selectable feature that allows you to control how documents are retrieved and ranked in your RAG system.

Quick Start

Get started with the retrieval system in 3 steps:

  1. Run Database Migration - Execute RETRIEVAL_SCHEMA.sql in your Supabase project
  2. Choose a Profile - Select from 5 predefined retrieval profiles
  3. Start Querying - Your queries will use the selected retrieval strategy

Architecture

The system is built on three layers:

  • Simple Profile Selector - Choose from predefined profiles without technical knowledge
  • Advanced Tuning Panel - Manually adjust weights and settings for power users
  • Evaluation Dashboard - View benchmark results and optimization improvements

Retrieval Profiles New

Profiles define how different retrieval methods are weighted and combined:

Built-in Profiles

  • Keyword Heavy - BM25: 50%, FTS: 30%, Vector: 20%
    Best for queries with specific terminology or technical terms
  • Balanced - BM25: 30%, FTS: 20%, Vector: 50%
    General-purpose profile, recommended default
  • Conceptual - BM25: 10%, FTS: 10%, Vector: 80%
    Emphasizes semantic similarity for conceptual queries
  • Academic Evidence - BM25: 25%, FTS: 25%, Vector: 50%
    Optimized for finding empirical evidence and research findings
  • Custom - User-defined weights
    Create your own profile with manual weight tuning

Hybrid Retrieval

The system combines three retrieval methods:

  1. BM25 - Traditional keyword-based ranking with TF-IDF weighting
  2. PostgreSQL FTS - Database-native full-text search with ranking
  3. Dense Vector Search - Semantic similarity using embeddings

Results are merged using weighted scoring or Reciprocal Rank Fusion (RRF).

Profile Selector

To set a retrieval profile for your project:

PUT /retrieval/projects/{collection}/profile
{
  "profile_name": "balanced"
}

Advanced Tuning

Power users can manually adjust retrieval weights:

PUT /retrieval/projects/{collection}/profile
{
  "bm25_weight": 0.3,
  "fts_weight": 0.2,
  "vec_weight": 0.5,
  "use_reranker": false
}

Benchmark Queries

Create evaluation queries to test retrieval performance:

POST /retrieval/benchmarks
{
  "collection": "my_collection",
  "query_text": "What is academic self-concept?",
  "query_type": "definitional",
  "source": "hand_curated"
}

Query Types

  • definitional - "What is X?", "Define Y"
  • empirical_evidence - "What research shows...", "What findings..."
  • critique - "What are the limitations...", "What problems..."
  • methodological - "How to measure...", "What methods..."
  • synthesis - "How do these relate...", "What is the overall picture..."

Relevance Labeling

Annotate which documents/chunks are relevant for each query:

POST /retrieval/labels
{
  "query_id": "query_123",
  "collection": "my_collection",
  "doc_id": "doc_456",
  "chunk_index": 5,
  "chunk_relevance": "supporting_evidence",
  "evidence_role": "definition",
  "annotator": "researcher_1"
}

Relevance Levels

  • Document Level: highly_relevant, somewhat_relevant, not_relevant
  • Chunk Level: supporting_evidence, background_only, not_relevant
  • Evidence Role: definition, background, empirical_finding, critique, method, limitation, gap

Evaluation Metrics

The system tracks comprehensive retrieval metrics:

  • Precision@K - Fraction of top-K results that are relevant
  • Recall@K - Fraction of relevant documents found in top-K
  • NDCG@K - Normalized Discounted Cumulative Gain (ranking quality)
  • MRR - Mean Reciprocal Rank (position of first relevant result)
  • MAP - Mean Average Precision (average precision across queries)

Grid Search Optimization

Automatically find the best weight combination:

  1. Create 20-30 benchmark queries
  2. Annotate relevance labels
  3. Run grid search across weight combinations
  4. System evaluates each combination and selects the best
  5. Apply winning profile to your project

Per-Query-Type Optimization

Different query types can use different optimal profiles:

  • Definitional queries may favor keyword matching
  • Conceptual queries may favor semantic search
  • System can automatically recommend profiles based on query type

Evaluation Dashboard

View optimization results:

GET /retrieval/runs?collection=my_collection
GET /retrieval/runs/best?collection=my_collection&metric=ndcg_at_10
GET /retrieval/optimizations?collection=my_collection

Profile Management API

GET    /retrieval/profiles              # List all profiles
GET    /retrieval/profiles/{name}       # Get specific profile
POST   /retrieval/profiles              # Create custom profile
PUT    /retrieval/profiles/{name}       # Update profile
DELETE /retrieval/profiles/{name}       # Delete custom profile

Benchmarks & Labels API

GET  /retrieval/benchmarks               # List queries
POST /retrieval/benchmarks               # Create query
POST /retrieval/benchmarks/bulk          # Bulk create

POST /retrieval/labels                   # Create label
POST /retrieval/labels/bulk              # Bulk create
GET  /retrieval/labels/{query_id}        # Get labels

Evaluation & Runs API

GET /retrieval/runs                      # List runs
GET /retrieval/runs/best                 # Get best run
GET /retrieval/optimizations             # List optimizations

Setup Guide

  1. Database Setup
    • Open Supabase Studio SQL Editor
    • Run RETRIEVAL_SCHEMA.sql
    • Verify tables created successfully
  2. Choose Default Profile
    • Start with "Balanced" profile
    • Test with sample queries
    • Adjust based on results
  3. Create Benchmarks
    • Collect 20-30 real user questions
    • Add hand-curated queries for key use cases
    • Generate from document section headings

Annotation Guide

  1. Start Annotation Session 
    POST /retrieval/annotations/sessions
{
  "collection": "my_collection",
  "annotator": "your_name"
}
  2. Review Retrieved Results
    • Run query with current profile
    • Review top 10 results
    • Mark relevant documents/chunks
  3. Label Relevance
    • Document level: Is this document useful?
    • Chunk level: Does this passage answer the query?
    • Evidence role: What type of evidence is this?

Optimization Guide

  1. Prepare Benchmark Set
    • Minimum 20 annotated queries
    • Cover different query types
    • Include easy, medium, and hard queries
  2. Run Grid Search
    • System tests weight combinations
    • Evaluates against benchmark metrics
    • Identifies best performing configuration
  3. Review Results
    • Check improvement over baseline
    • Review per-query-type performance
    • Validate on held-out queries
  4. Apply Best Profile
    • One-click application to project
    • Monitor production performance
    • Iterate based on user feedback

For more detailed information, see the full guides in the project repository.
